# บทที่ 0 — ติดตั้งระบบ และตั้งผู้ดูแลระบบ

บทนี้สำหรับ **ผู้ดูแลระบบ (System Administrator / IT)** ที่ติดตั้ง TEAS ครั้งแรก
ตั้งแต่เครื่องเปล่า จนถึงพร้อมให้ผู้ใช้ทั่วไปเข้าใช้งาน. ลำดับ:

1. เตรียมเครื่อง (ความต้องการระบบ)
2. ติดตั้งและตั้งค่า (database, JWT, build + run)
3. เริ่มระบบครั้งแรก (ฐานข้อมูลตั้งต้นอัตโนมัติ)
4. เข้าสู่ระบบด้วยผู้ดูแลสูงสุดคนแรก (bootstrap super admin)
5. ตั้งค่าครั้งแรก: สร้างบริษัทแรกผ่าน setup wizard + การสลับบริษัท
6. รายการตรวจก่อนใช้งานจริง (pre-go-live checklist)

> ขั้นที่ 0.1–0.4 เป็นงานเชิงเทคนิค (terminal/command) — ส่วนใหญ่ไม่มีภาพหน้าจอ
> เพราะเป็นขั้นก่อนมีหน้าเว็บให้ใช้. ภาพหน้าจอจริงเริ่มตั้งแต่ขั้นที่ 0.4
> (เข้าสู่ระบบ) เป็นต้นไป ในบทถัด ๆ ไป.

## 0.1 ความต้องการระบบ (Prerequisites)

| ส่วนประกอบ | เวอร์ชันขั้นต่ำ | หมายเหตุ |
|---|---|---|
| **.NET SDK** | 10.0 LTS | Backend (ASP.NET Core Minimal API + EF Core 10) |
| **Node.js** | 20 LTS+ | Frontend (Next.js 15) — ใช้ `pnpm` หรือรัน Next ผ่าน `node` โดยตรง |
| **PostgreSQL** | 16+ | ฐานข้อมูล (ระบบเปิด extension ที่จำเป็น เช่น `pgcrypto`, `uuid-ossp` ให้อัตโนมัติ) |
| **OS (ฝั่ง server)** | Windows / Linux | ทดสอบบน Windows 11 และ Linux |
| **Browser (ฝั่งผู้ใช้)** | Chrome 78+ / Edge 18+ / Firefox 70+ / Safari 13+ | ดูบทที่ 1 |

**ทรัพยากรแนะนำ:** เริ่มต้นที่ 2 vCPU / 4 GB RAM สำหรับ instance ขนาดเล็ก
และพื้นที่ดิสก์เผื่อฐานข้อมูล + ไฟล์เอกสาร PDF/XML. เพิ่มตามจำนวนผู้ใช้
พร้อมกันและปริมาณเอกสารที่ออกต่อเดือน.

> **เวลาเครื่อง (timezone):** ระบบเก็บเวลาภายในเป็น UTC (`DateTimeOffset`) และ
> แปลงเป็น `Asia/Bangkok` เฉพาะตอนแสดงผล. ตั้งนาฬิกาเครื่อง server ให้ตรง
> (sync NTP) เพราะ tax-point date ของเอกสารอ้างวันที่ปัจจุบันตามเขตเวลาไทย.

## 0.2 การติดตั้ง (Installation)

### ขั้นที่ 1 — สร้างฐานข้อมูล

สร้างฐานข้อมูลและผู้ใช้ฐานข้อมูล (ตัวอย่างค่าเริ่มต้นสำหรับ development):

```sql
CREATE DATABASE accounting_dev;
CREATE USER accounting WITH PASSWORD 'เปลี่ยนรหัสนี้';
GRANT ALL PRIVILEGES ON DATABASE accounting_dev TO accounting;
```

> **ความปลอดภัย:** ในระบบจริง (production) ตั้งรหัสผ่านฐานข้อมูลที่คาดเดายาก
> เก็บใน secret manager และห้าม commit ลง git. แนะนำให้ฐานข้อมูลรับ
> connection จากเฉพาะ host ของ backend เท่านั้น.

### ขั้นที่ 2 — ตั้งค่าการเชื่อมต่อและคีย์ลับ

Backend อ่านค่าจาก environment variable / `appsettings`. ค่าหลักที่ต้องตั้ง:

```
ConnectionStrings__Default = Host=localhost;Port=5432;Database=accounting_dev;Username=accounting;Password=...
ASPNETCORE_ENVIRONMENT      = Development          # หรือ Production
ASPNETCORE_URLS             = http://localhost:5080
Jwt__Issuer                 = teas                 # ชื่อผู้ออก token
Jwt__Audience               = teas                 # ผู้รับ token
Jwt__SigningKey             = <คีย์ลับยาว ≥ 32 ตัวอักษร>   # ใช้ลงนาม JWT
```

> **คีย์ลงนาม JWT (`Jwt__SigningKey`)** เป็นความลับสำคัญ — ใครได้คีย์นี้ไป
> สามารถปลอม token เข้าระบบได้. สร้างคีย์สุ่มยาว เก็บใน secret manager
> และใช้คีย์คนละชุดระหว่าง dev / production. ห้าม commit ลง git.

Frontend (`.env.local`) ชี้ไปที่ backend ผ่าน BFF proxy:

```
NEXT_PUBLIC_API_BASE = http://localhost:5080
```

> JWT จะถูกเก็บใน **httpOnly cookie** ฝั่ง BFF — JavaScript ในหน้าเว็บอ่านไม่ได้
> (กัน token ถูกขโมยจาก XSS).

### ขั้นที่ 3 — Build และรัน

**Backend** (จากโฟลเดอร์ `backend`):

```bash
dotnet build Accounting.sln
# รัน API ที่ http://localhost:5080 (ต้องตั้ง ASPNETCORE_ENVIRONMENT=Development)
dotnet run --project src/Accounting.Api
```

> **สำคัญ:** ใน development ต้องรันด้วย `ASPNETCORE_ENVIRONMENT=Development`
> มิฉะนั้น login จะ error. API จะ migrate + seed ฐานข้อมูลให้อัตโนมัติเมื่อ
> เริ่มครั้งแรก (ดู 0.3).

**Frontend** (จากโฟลเดอร์ `frontend`):

```bash
pnpm install
pnpm build      # production build
pnpm start      # เปิดที่ http://localhost:3000
# ระหว่างพัฒนา ใช้ pnpm dev แทน
```

เมื่อทั้งสองฝั่งรันแล้ว เปิด browser ไปที่ `http://localhost:3000` จะเจอหน้า login.

## 0.3 การเริ่มต้นฐานข้อมูลครั้งแรก (First DB Init)

เมื่อ Backend เริ่มทำงานครั้งแรก ตัวระบบ (`DbInitializer`) จะทำให้ฐานข้อมูล
พร้อมใช้งานโดยอัตโนมัติ — **ไม่ต้องรันคำสั่ง migration เองด้วยมือ**:

1. **รัน EF Core migrations** — สร้าง schema ทั้งหมด (ตาราง, index, foreign key, constraint)
2. **รัน SQL seed scripts** — ใส่ข้อมูลตั้งต้นและกลไกความปลอดภัย:
   - **roles + permissions** — ชุด role มาตรฐานและสิทธิ์ (RBAC) ทั้งหมด
   - **document prefixes** — รหัสนำหน้าเลขเอกสาร (TI, RC, CN, PV, ฯลฯ) สำหรับการรันเลขแบบไม่มีช่องว่าง
   - **ผังบัญชีตัวอย่าง (chart of accounts)** — โครงผังบัญชีตั้งต้น
   - **ประเภทหัก ณ ที่จ่ายมาตรฐาน (WHT types)** และ **หมวดค่าใช้จ่าย**
   - **trigger ความ immutable** — กันการแก้/ลบเอกสารที่ post แล้วที่ระดับฐานข้อมูล (มาตรา §4.2)
   - **PostgreSQL Row-Level Security (RLS)** — แยกข้อมูลตาม `company_id` ทุกตารางธุรกิจ (มาตรา §4.7)
3. **บันทึกว่า script ใดรันแล้ว** ในตาราง `sys.applied_sql_scripts` — แต่ละ script
   รัน **ครั้งเดียว** (idempotent) จึงเริ่ม API ซ้ำได้โดยไม่ทำข้อมูลซ้ำ

> Log จะแสดง `Apply SQL script: ...` สำหรับ script ที่เพิ่งรัน และ
> `Skip ... already applied` สำหรับ script ที่รันไปแล้ว. หากเห็น error ระหว่างนี้
> ให้แก้ก่อนใช้งาน — อย่ารันเอกสารจริงบนฐานข้อมูลที่ seed ไม่ครบ.

## 0.4 ผู้ดูแลระบบสูงสุด (Bootstrap Super Admin)

ระบบสร้างผู้ดูแลสูงสุด (Super Admin) คนแรกให้อัตโนมัติจาก seed:

| ฟิลด์ | ค่าเริ่มต้น |
|---|---|
| ชื่อผู้ใช้ (username) | `admin` |
| รหัสผ่าน (password) | `Admin@1234` |
| สิทธิ์ | **Super Admin** (`is_super_admin`) — ข้ามทุกบริษัท, ทุกสิทธิ์ |

**Super Admin คืออะไร:** เป็น flag ระดับบัญชีผู้ใช้ (`is_super_admin`) ที่ให้สิทธิ์
**ข้ามทุกบริษัท** — สร้างบริษัท, ตั้งค่าภาษีระดับบริษัท, และสลับเข้าดูแลทุก
บริษัทในระบบได้. ต่างจากผู้ดูแล **ระดับบริษัท** (Company Admin) ที่จัดการ
role/ผู้ใช้ได้เฉพาะภายในบริษัทตัวเอง.

### เข้าสู่ระบบครั้งแรก

ไปที่ `http://localhost:3000/login` แล้วเข้าสู่ระบบด้วยบัญชี `admin`
(ดูวิธีโดยละเอียดในบทที่ 1 — เข้าสู่ระบบ).

> ### ⚠️ ความปลอดภัย — ทำทันทีหลังติดตั้ง
>
> 1. **เปลี่ยนรหัสผ่าน `admin` ทันที** เป็นรหัสที่คาดเดายาก ก่อนเปิดให้ผู้อื่นเข้าถึงระบบ.
> 2. อย่าใช้บัญชี `admin` ทำงานประจำวัน — สร้างผู้ใช้แยกตามหน้าที่ (ดู 0.6).
> 3. เก็บบัญชี Super Admin ไว้เท่าที่จำเป็น และทบทวนเป็นระยะ.

### เพิ่ม Super Admin คนใหม่

เพราะ Super Admin ข้ามทุกบริษัท จึง **กำหนดผ่านหน้าจัดการผู้ใช้ปกติไม่ได้**
(โดยตั้งใจ — กันการยกสิทธิ์ข้ามบริษัทโดยพลาด). การเพิ่ม Super Admin คนใหม่
เป็นงานระดับฐานข้อมูล/ops:

```sql
UPDATE sys.users SET is_super_admin = TRUE WHERE username = 'ชื่อผู้ใช้';
```

## 0.5 ตั้งค่าครั้งแรก: สร้างบริษัทแรก (First-time Setup / Onboarding)

หลังเปลี่ยนรหัสผ่าน Super Admin แล้ว ขั้นต่อไปคือ **สร้างบริษัทแรก**.
ระบบรองรับหลายบริษัทในติดตั้งเดียว (multi-tenant) — ทุกเอกสารผูกกับ
`company_id` ของบริษัทที่ session กำลังใช้งาน.

### 0.5.1 Setup wizard — บริษัทแรก

เมื่อ Super Admin เข้าสู่ระบบและ **ยังไม่มีบริษัทผูกกับบัญชี** (สถานะ
"ยังไม่มีบริษัท" — ระบบดูจากการที่บัญชียังไม่ถูกมอบหมายให้บริษัทใด ไม่ใช่
จากการที่ฐานข้อมูลว่างเปล่า) ระบบจะนำเข้าสู่ **หน้า setup wizard** เพื่อ
สร้างบริษัทแรกก่อนเข้าใช้งานหน้าอื่น.

ข้อมูลที่ wizard ให้กรอกเพื่อสร้างบริษัท:

- **ชื่อนิติบุคคล (ไทย / อังกฤษ)**
- **เลขประจำตัวผู้เสียภาษี 13 หลัก**
- **สถานะการจดทะเบียน VAT** — จด / ไม่จด, **อัตรา VAT** (เช่น 7%),
  และ **โหมดยื่น ภ.พ.30** (อัตโนมัติผ่าน RD API หรือ ด้วยตนเอง)
- **ปีบัญชี (fiscal year)**
- **สกุลเงิน (currency)**

เมื่อสร้างเสร็จ ระบบจะพาเข้าสู่ **Dashboard ของบริษัทนั้น** พร้อมเริ่ม
ตั้งค่าข้อมูลหลักและออกเอกสารได้ทันที.

> _(ภาพหน้าจอ wizard จะเพิ่มหลังฟีเจอร์ onboarding ขึ้นจริง)_

> ### ทำไมการตั้งค่าภาษีอยู่ที่ Super Admin เท่านั้น (มาตรา §4.6)
>
> สถานะ VAT / อัตรา / โหมด ภ.พ.30 เป็น **ข้อมูลหลักระดับบริษัท** ที่กระทบ
> ความถูกต้องตามกฎหมายของทุกเอกสาร — จึงตั้งและแก้ได้เฉพาะ Super Admin
> และทุกครั้งที่แก้ฟิลด์ภาษีจะถูกบันทึกลง audit log. ผู้ใช้ทั่วไป
> **ไม่มี** หน้าตั้งค่า VAT.

> บริษัทที่ไม่จด VAT จะ **ซ่อน** เมนู/ปุ่มที่เกี่ยวกับ VAT โดยอัตโนมัติ
> (ใบกำกับภาษี, ใบลดหนี้/เพิ่มหนี้, ภ.พ.30) — ดู *คู่มือสิทธิ์การมองเห็น (RBAC UI Guide)*.

### 0.5.2 สลับบริษัท (Company Switcher)

เมื่อมีหลายบริษัทในระบบ Super Admin สามารถ **สลับบริษัทที่ใช้งานอยู่**
(active company) ได้ผ่าน **dropdown สลับบริษัท** ที่แถบบนของหน้าจอ.
เลือกบริษัทจากรายการ → ระบบจะเปลี่ยน context ไปยังบริษัทนั้น (ข้อมูล,
เอกสาร, รายงาน ทั้งหมดจะเป็นของบริษัทที่เลือก). ตัวสลับบริษัทนี้
**เห็นเฉพาะ Super Admin** — ผู้ใช้ทั่วไปที่ผูกกับบริษัทเดียวจะไม่เห็น.

> _(ภาพหน้าจอ company switcher จะเพิ่มหลังฟีเจอร์ขึ้นจริง)_

### 0.5.3 สร้างบริษัทเพิ่มในภายหลัง

Super Admin สร้างบริษัทเพิ่มได้เสมอ (ไม่จำกัดที่บริษัทแรก) ด้วยข้อมูลชุด
เดียวกับ wizard ข้างต้น. **สิ่งที่เกิดอัตโนมัติเมื่อสร้างบริษัทใหม่:** ระบบ
clone ชุด role มาตรฐานให้บริษัทใหม่ทันที (Company Admin, Chief Accountant,
Accountant, AR/AP Clerk, Sales/Purchasing/Warehouse Staff, Approver, Auditor,
Tax Officer) พร้อมสิทธิ์ตามหน้าที่ — พร้อมมอบหมายให้ผู้ใช้ได้เลย.

## 0.6 รายการตรวจก่อนใช้งานจริง (Pre-Go-Live Checklist)

หลังสร้างบริษัทแล้ว ทำรายการต่อไปนี้ให้ครบก่อนเริ่มออกเอกสารจริง.
ลำดับเรียงตามความสำคัญ — ข้อ ☑ คือทำเสร็จแล้วในบทนี้.

- ☑ **ติดตั้ง + เริ่มระบบ + เปลี่ยนรหัส `admin`** (0.2–0.4)
- ☑ **สร้างบริษัท + ตั้งค่าภาษี (VAT / อัตรา / โหมด ภ.พ.30)** (0.5)
- ☐ **ข้อมูลบริษัทสำหรับหัวเอกสาร (Company Profile)** — ชื่อนิติบุคคล, ที่อยู่
  จดทะเบียน, เลขผู้เสียภาษี, รหัสสาขา, โลโก้ ให้ตรงกับ ภ.พ.20 ของกรมสรรพากร.
  ข้อมูลนี้ถูก embed ในใบกำกับภาษีทุกใบ (มาตรา §4.1 / ม.86/4) — **ทำก่อน
  ออกเอกสารใบแรก** *(บทที่ 2)*
- ☐ **ทบทวนผังบัญชี (Chart of Accounts)** — ปรับโครงผังบัญชีที่ seed มาให้ตรง
  กับธุรกิจ และตรวจ **บัญชีตั้งต้นด้านภาษี**: VAT ขายต้องชำระ (output VAT
  payable), VAT ซื้อขอคืน (input VAT receivable), และบัญชีพักภาษีหัก ณ ที่จ่าย
  (WHT) — ใช้ลงบัญชีอัตโนมัติเมื่อออกเอกสาร
- ☐ **ตั้งค่าประเภทหัก ณ ที่จ่าย (WHT types)** — ประเภทเงินได้ + อัตรา + แบบยื่น
  (ภ.ง.ด.3/53/54) ตาม ม.40 ของประมวลรัษฎากร *(บทที่ 2)*
- ☐ **รหัสนำหน้าเลขเอกสาร (Document number prefixes)** — ตรวจรูปแบบ
  `MM-YYYY-PREFIX-NNNN` (PV เพิ่ม `-CATEGORY-`) ให้ตรงกับที่ต้องการ.
  เลขต้องรันต่อเนื่องไม่มีช่องว่าง รีเซ็ตรายเดือน (มาตรา §4.3)
- ☐ **หน่วยธุรกิจ (Business Units)** — กำหนดรหัส BU (3–4 ตัวอักษร) ที่จะแทรก
  ในเลขเอกสารเพื่อแยกชุด เช่น `12-2026-TI-ECOM-0001` *(บทที่ 2)*
- ☐ **ข้อมูลหลักอื่น** — สินค้า/บริการ (พร้อมประเภท VAT ให้ถูก), หมวดค่าใช้จ่าย,
  พนักงาน *(บทที่ 2)*
- ☐ **สร้างผู้ใช้ + มอบหมาย role ตามหน้าที่** — Super Admin หรือ Company Admin
  ใช้หน้า **ตั้งค่า → ผู้ใช้** สร้าง/มอบหมาย role (หลาย role ต่อผู้ใช้ได้), และ
  **ตั้งค่า → บทบาท** ปรับชุดสิทธิ์ของ role ในบริษัทตัวเอง (role มาตรฐานที่เป็น
  *ระบบ* ลบไม่ได้). ดูสรุปสิทธิ์ใน *คู่มือสิทธิ์การมองเห็น (RBAC UI Guide)*

> **แยกหน้าที่ (Segregation of Duties):** มอบ "ผู้สร้างเอกสาร" และ "ผู้อนุมัติ"
> ให้คนละคน เช่น ให้ AR Clerk ออกใบกำกับภาษี แต่ให้ Approver / Chief Accountant
> เป็นผู้อนุมัติ/โพสต์. ระบบบังคับสิทธิ์ราย endpoint ที่ฝั่ง backend ทุกครั้ง
> (UI แค่ซ่อนปุ่มที่ใช้ไม่ได้เป็นการอำนวยความสะดวก — ไม่ใช่ขอบเขตความปลอดภัยเอง).

> เมื่อทำครบทั้งหมดข้างต้น ระบบพร้อมให้ผู้ใช้ทั่วไปเข้าใช้งาน —
> อ่านต่อ **บทที่ 1 เริ่มต้นใช้งาน** และ **บทที่ 2 ตั้งค่าระบบ**.
