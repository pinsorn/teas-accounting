# บทที่ 0 — ติดตั้งระบบ และตั้งผู้ดูแลระบบ

บทนี้สำหรับ **ผู้ดูแลระบบ (System Administrator / IT)** ที่ติดตั้ง TEAS ครั้งแรก
ตั้งแต่เครื่องเปล่า จนถึงพร้อมให้ผู้ใช้ทั่วไปเข้าใช้งาน. ลำดับ:

1. เตรียมเครื่อง (ความต้องการระบบ)
2. ติดตั้งและตั้งค่า
3. เริ่มระบบครั้งแรก (ฐานข้อมูลตั้งต้นอัตโนมัติ)
4. เข้าสู่ระบบด้วยผู้ดูแลสูงสุดคนแรก (bootstrap super admin)
5. สร้างบริษัท (พร้อมตั้งค่าภาษี)
6. สร้าง role และผู้ใช้
7. ลำดับการตั้งค่าที่แนะนำก่อนเปิดใช้งานจริง

> เอกสารส่วนนี้เป็นขั้นตอนเชิงเทคนิค (terminal/command) — ไม่มีภาพหน้าจอ
> เพราะยังไม่มีหน้าเว็บให้บันทึก. ภาพหน้าจอจริงเริ่มตั้งแต่ขั้นที่ 4 (เข้าสู่ระบบ)
> เป็นต้นไป ในบทถัด ๆ ไป.

## 0.1 ความต้องการระบบ (Prerequisites)

| ส่วนประกอบ | เวอร์ชันขั้นต่ำ | หมายเหตุ |
|---|---|---|
| **.NET SDK** | 10.0 LTS | Backend (ASP.NET Core Minimal API) |
| **Node.js** | 20 LTS+ | Frontend (Next.js 15) — ใช้ `pnpm` หรือ `node` รัน Next โดยตรง |
| **PostgreSQL** | 16+ | ฐานข้อมูล (เปิด extension `pgcrypto`, `uuid-ossp` อัตโนมัติ) |
| **OS** | Windows / Linux | ระบบทดสอบบน Windows 11 + Linux |
| Browser (ฝั่งผู้ใช้) | Chrome/Edge/Firefox รุ่นใหม่ | ดูบทที่ 1 |

**ทรัพยากรแนะนำ:** 2 vCPU / 4 GB RAM ขึ้นไปสำหรับ instance ขนาดเล็ก
(เพิ่มตามจำนวนผู้ใช้พร้อมกันและปริมาณเอกสาร).

## 0.2 ติดตั้งและตั้งค่า

### ขั้นที่ 1 — เตรียมฐานข้อมูล

สร้างฐานข้อมูลและผู้ใช้ฐานข้อมูล (ตัวอย่างค่าเริ่มต้นสำหรับ development):

```sql
CREATE DATABASE accounting_dev;
CREATE USER accounting WITH PASSWORD 'เปลี่ยนรหัสนี้';
GRANT ALL PRIVILEGES ON DATABASE accounting_dev TO accounting;
```

> **ความปลอดภัย:** ในระบบจริง (production) ตั้งรหัสผ่านฐานข้อมูลที่คาดเดายาก
> และเก็บใน secret manager — ห้าม commit ลง git.

### ขั้นที่ 2 — ตั้งค่าการเชื่อมต่อ

Backend อ่าน connection string จาก environment / `appsettings`. ค่าหลักที่ต้องตั้ง:

```
ConnectionStrings__Default = Host=localhost;Port=5432;Database=accounting_dev;Username=accounting;Password=...
ASPNETCORE_ENVIRONMENT      = Development        # หรือ Production
ASPNETCORE_URLS             = http://localhost:5080
Jwt__Issuer / Jwt__Audience / Jwt__SigningKey   # คีย์ลงนาม JWT (ตั้งคีย์ลับยาว ≥ 32 ตัวอักษร)
```

Frontend (`.env.local`) ชี้ไปที่ backend ผ่าน BFF proxy:

```
NEXT_PUBLIC_API_BASE = http://localhost:5080
```

> JWT จะถูกเก็บใน **httpOnly cookie** ฝั่ง BFF — JavaScript ในหน้าเว็บอ่านไม่ได้
> (กัน token ถูกขโมยจาก XSS).

### ขั้นที่ 3 — Build และรัน

**Backend** (จากโฟลเดอร์ `backend`):

```bash
dotnet build Accounting.sln
# รัน API (Development)
dotnet run --project src/Accounting.Api
```

**Frontend** (จากโฟลเดอร์ `frontend`):

```bash
pnpm install
pnpm build      # production build
pnpm start      # เปิดที่ http://localhost:3000
# ระหว่างพัฒนา ใช้ pnpm dev แทน
```

## 0.3 เริ่มระบบครั้งแรก — ฐานข้อมูลตั้งต้นอัตโนมัติ

เมื่อ Backend เริ่มทำงานครั้งแรก ตัวระบบ (`DbInitializer`) จะทำให้ฐานข้อมูล
พร้อมใช้งานโดยอัตโนมัติ:

1. **รัน EF Core migrations** — สร้าง schema ทั้งหมด (ตาราง, index, foreign key)
2. **รัน SQL seed scripts** — ใส่ข้อมูลตั้งต้น: roles + permissions, document
   prefixes, ผังบัญชีตัวอย่าง, ประเภทหัก ณ ที่จ่ายมาตรฐาน, หมวดค่าใช้จ่าย,
   trigger ความ immutable ของเอกสารที่ post แล้ว, PostgreSQL Row-Level Security
3. **บันทึกว่าscript ใดรันแล้ว** ในตาราง `sys.applied_sql_scripts` — แต่ละ script
   รัน **ครั้งเดียว** (idempotent) จึงเริ่ม API ซ้ำได้โดยไม่ทำข้อมูลซ้ำ

> ไม่ต้องรันคำสั่ง migration เองด้วยมือ — แค่เริ่ม API. Log จะแสดง
> `Apply SQL script: ...` สำหรับ script ที่เพิ่งรัน และ `Skip ... already applied`
> สำหรับ script ที่รันไปแล้ว.

## 0.4 ผู้ดูแลระบบสูงสุดคนแรก (Bootstrap Super Admin)

ระบบสร้างผู้ดูแลสูงสุด (Super Admin) คนแรกให้อัตโนมัติจาก seed:

| ฟิลด์ | ค่าเริ่มต้น |
|---|---|
| ชื่อผู้ใช้ (username) | `admin` |
| รหัสผ่าน (password) | `Admin@1234` |
| บริษัท | บริษัทตั้งต้น (company 1) |
| สิทธิ์ | **Super Admin** (`is_super_admin`) — ข้ามทุกบริษัท, ทุกสิทธิ์ |

ลองเข้าสู่ระบบที่ `http://localhost:3000/login` ด้วยบัญชีนี้ (ดูบทที่ 1 — เข้าสู่ระบบ).

> ### ⚠️ ความปลอดภัย — ทำทันทีหลังติดตั้ง
>
> 1. **เปลี่ยนรหัสผ่าน `admin` ทันที** เป็นรหัสที่คาดเดายาก.
> 2. อย่าใช้บัญชี `admin` ทำงานประจำวัน — สร้างผู้ใช้แยกตามหน้าที่ (ขั้นที่ 0.6).
> 3. เก็บบัญชี Super Admin ไว้เท่าที่จำเป็น.

### Super Admin เพิ่มเติม

สิทธิ์ **Super Admin** คือ flag ระดับบัญชีผู้ใช้ (`is_super_admin`) ที่ข้ามทุกบริษัท
จึง **กำหนดผ่านหน้าจัดการผู้ใช้ไม่ได้** (โดยตั้งใจ — กันการยกสิทธิ์ข้ามบริษัทโดยพลาด).
การเพิ่ม Super Admin คนใหม่เป็นงานระดับฐานข้อมูล/ops:

```sql
UPDATE sys.users SET is_super_admin = TRUE WHERE username = 'ชื่อผู้ใช้';
```

สำหรับผู้ดูแล **ระดับบริษัท** (Company Admin) ที่จัดการ role/ผู้ใช้ภายในบริษัทตัวเอง
ให้ใช้หน้าจอ RBAC ปกติ (ขั้นที่ 0.6) — ไม่ต้องแตะฐานข้อมูล.

## 0.5 สร้างบริษัท (พร้อมตั้งค่าภาษี)

เข้าสู่ระบบเป็น Super Admin → เมนู **ตั้งค่า → บริษัท (Companies)**
(`/settings/companies`, เห็นเฉพาะ Super Admin) → **เพิ่มบริษัท**.

ข้อมูลที่ต้องกรอก:

- **ชื่อนิติบุคคล (ไทย/อังกฤษ)**, ประเภทนิติบุคคล
- **เลขประจำตัวผู้เสียภาษี 13 หลัก** + รหัสสาขา (`00000` = สำนักงานใหญ่)
- **การจดทะเบียน VAT**: จด/ไม่จด, อัตรา VAT (เช่น 7%), วันที่จดทะเบียน
- **โหมดยื่น ภ.พ.30**: อัตโนมัติ (auto, ผ่าน RD API) หรือ ด้วยตนเอง (manual)

> ### ทำไมการตั้งค่าภาษีอยู่ที่ Super Admin เท่านั้น (มาตรา §4.6)
>
> สถานะ VAT / อัตรา / โหมด ภ.พ.30 เป็น **ข้อมูลหลักระดับบริษัท** ที่กระทบ
> ความถูกต้องตามกฎหมายของทุกเอกสาร — จึงแก้ได้เฉพาะ Super Admin และทุกครั้ง
> ที่แก้ฟิลด์ภาษีจะถูกบันทึกลง audit log. ผู้ใช้ทั่วไป **ไม่มี** หน้าตั้งค่า VAT.

**สิ่งที่เกิดอัตโนมัติเมื่อสร้างบริษัท:** ระบบ clone ชุด role มาตรฐาน 11 role
(Company Admin, Chief Accountant, Accountant, AR/AP Clerk, Sales/Purchasing/
Warehouse Staff, Approver, Auditor, Tax Officer) พร้อมสิทธิ์ตามหน้าที่ ให้บริษัทใหม่ทันที
— พร้อมมอบหมายให้ผู้ใช้ได้เลย.

> บริษัทที่ไม่จด VAT จะ **ซ่อน** เมนู/ปุ่มที่เกี่ยวกับ VAT โดยอัตโนมัติ
> (ใบกำกับภาษี, ใบลดหนี้/เพิ่มหนี้, ภ.พ.30) — ดู *คู่มือสิทธิ์การมองเห็น (RBAC UI Guide)*.

## 0.6 สร้าง role และผู้ใช้

หลังมีบริษัทแล้ว Super Admin หรือ Company Admin ของบริษัทนั้นจัดการสิทธิ์ผ่านหน้าจอได้:

- **ตั้งค่า → ผู้ใช้ (`/settings/users`)** — สร้าง/แก้ผู้ใช้ และมอบหมาย role
  (เลือกได้หลาย role ต่อผู้ใช้). ต้องมีสิทธิ์ `sys.user.manage`.
- **ตั้งค่า → บทบาท (`/settings/roles`)** — สร้าง role เอง หรือปรับชุดสิทธิ์ของ role
  ในบริษัทตัวเอง (role มาตรฐานที่เป็น *ระบบ* ลบไม่ได้). ต้องมีสิทธิ์ `sys.role.manage`.

> **แยกหน้าที่ (Segregation of Duties):** มอบ "ผู้สร้างเอกสาร" และ "ผู้อนุมัติ"
> ให้คนละคน เช่น ให้ AR Clerk ออกใบกำกับภาษี แต่ให้ Approver/Chief Accountant
> เป็นผู้อนุมัติ/โพสต์. ระบบบังคับสิทธิ์ราย endpoint ทุกครั้งที่ฝั่ง backend
> (UI แค่ซ่อนปุ่มที่ใช้ไม่ได้เป็นการอำนวยความสะดวก — ไม่ใช่ขอบเขตความปลอดภัยเอง).

ใครเห็น/ใช้อะไรได้บ้าง ดูสรุปแบบตารางใน *คู่มือสิทธิ์การมองเห็น (RBAC UI Guide)*.

## 0.7 ลำดับการตั้งค่าที่แนะนำ ก่อนเปิดใช้งานจริง

1. ✅ ติดตั้ง + เริ่มระบบ + เปลี่ยนรหัส `admin` (บทนี้ 0.2–0.4)
2. ✅ สร้างบริษัท + ตั้งค่าภาษี (0.5)
3. **ตั้งค่าข้อมูลบริษัท (Company Profile)** — ชื่อ/ที่อยู่/โลโก้ ที่จะขึ้นหัวเอกสารทุกใบ
   *(ทำก่อนออกเอกสารใบแรก — บทที่ 2)*
4. สร้างผู้ใช้ + มอบหมาย role ตามหน้าที่ (0.6)
5. ตั้งค่าข้อมูลหลัก: ผังบัญชี, หน่วยธุรกิจ, สินค้า/บริการ, ประเภทหัก ณ ที่จ่าย,
   หมวดค่าใช้จ่าย, พนักงาน *(บทที่ 2)*
6. เริ่มงานเอกสาร: ขาย (ใบเสนอราคา → ใบกำกับภาษี → ใบเสร็จ) / ซื้อ (ใบสั่งซื้อ →
   ใบสำคัญจ่าย) *(บทถัด ๆ ไป)*

> เมื่อทำขั้นที่ 1–4 เสร็จ ระบบพร้อมให้ผู้ใช้ทั่วไปเข้าใช้งาน — อ่านต่อ **บทที่ 1
> เริ่มต้นใช้งาน**.
