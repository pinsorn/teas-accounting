  ของเลขเอกสาร เช่น `12-2026-TI-ECOM-0001`)

## แนวคิดที่ควรรู้

### ข้อมูลบริษัท — Hybrid lock model

ระบบแยก fields เป็น 2 ระดับ (plan §6.7):

**Hard fields** (อ่านอย่างเดียวใน Phase 1) — ข้อมูลตามทะเบียน VAT (ภ.พ.20):
ชื่อนิติบุคคล, เลขผู้เสียภาษี, เลขทะเบียนนิติบุคคล, ที่อยู่จดทะเบียน,
รหัสสาขา, วันที่จดทะเบียน VAT. ต้องแก้ผ่าน ops + ยื่น ภ.พ.09 ก่อน
(Phase 2 จะรองรับ 2-person approval + ภ.พ.09 attachment).

**Soft fields** (admin edit ได้): trade name, โลโก้, เบอร์/อีเมล/เว็บไซต์,
contact name, banking info.

### หน่วยธุรกิจ (Business Unit)

BU คือการแบ่งกลุ่มภายในบริษัทเดียวกัน — รหัส 3-4 ตัวอักษร (เช่น ECOM,
LAB, REPT) ที่จะถูกแทรกในเลขเอกสารเพื่อแยกชุด:
`12-2026-TI-ECOM-0001` vs `12-2026-TI-LAB-0001`.

ระบบรองรับ **บังคับเลือก BU** ในเอกสารรายได้ทุกประเภท (toggle ที่
ด้านบนของหน้า Business Unit) — ถ้าเปิด: ออก TI/RC/CN/DN ต้องระบุ BU ทุกครั้ง.

### ประเภทสินค้า/บริการ และ VAT

ระบบรองรับ 4 ประเภท ซึ่งกระทบการคิด VAT 7% ทันที:

| ประเภท | VAT 7% | ตัวอย่าง |
|---|---|---|
| GOOD | ✓ ต้องเสีย | ตู้เลี้ยงปลา, เครื่องกรองน้ำ |
| SERVICE | ✓ ต้องเสีย | ค่าที่ปรึกษา, ค่าบริการตรวจ |
| EXEMPT_GOOD | — ยกเว้น | สัตว์มีชีวิต, อาหารสัตว์ |
| EXEMPT_SERVICE | — ยกเว้น | บริการการศึกษา, ค่ารักษาพยาบาล |

การตั้งประเภทผิด → คิด VAT ผิด → ภ.พ.30 ยื่นผิด → โดนเบี้ยปรับ.

### ประเภทหัก ณ ที่จ่าย (WHT) + แบบยื่น

ระบบแยกอัตรา WHT ตาม **ประเภทเงินได้ (มาตรา 40)** และ **แบบยื่นภาษี**:

| แบบยื่น | ใช้กับ |
|---|---|
| ภ.ง.ด.1 | เงินเดือน, ค่าจ้างพนักงาน |
| ภ.ง.ด.3 | ค่าเช่า, ค่าวิชาชีพอิสระ (จ่ายให้บุคคล) |
| ภ.ง.ด.53 | ค่าบริการ, ค่าโฆษณา (จ่ายให้นิติบุคคล) |
| ภ.ง.ด.54 | จ่ายต่างประเทศ (foreign vendor) |

ปุ่ม **"เปลี่ยนอัตรา"** ใน row ของแต่ละประเภท — ใช้เมื่อกฎหมายปรับ
อัตราหัก (effective-date pattern, plan §16.4) — ระบบจะเก็บประวัติอัตรา
เดิมไว้พร้อม timestamp มีผล "ตั้งแต่ / ถึง" เพื่อใช้กับเอกสารย้อนหลัง.

### API Keys + Per-Key BU Binding

API Keys ใช้สำหรับให้ระบบภายนอก (microservice, mobile app, integration)
เรียก TEAS โดยไม่ต้องใช้ JWT/login. แต่ละ key มี:

- **Scopes**: granular permissions เช่น `sales.tax_invoice.create`,
  `sales.receipt.post` — ให้ผู้เรียกทำได้เฉพาะที่จำเป็น (least privilege)
- **Business Unit เริ่มต้น**: ผูก key กับ 1 BU เท่านั้น (Sprint 14)
  — เอกสารที่สร้างจาก key นี้จะถูก tag กับ BU นั้นอัตโนมัติ
- **หมดอายุ**: optional — แนะนำให้ตั้งสำหรับ contractor / temp staff

**🔐 Critical**: key เห็นได้ครั้งเดียวหลังกด "สร้าง" — copy เก็บใน secret
manager ทันที. ระบบเก็บแค่ hash + prefix.

## พฤติกรรมระบบที่ปลอดภัย — ที่ผู้ใช้ควรรู้

ทุก destructive action (ปิดใช้งาน BU/Product) จะแสดง **AlertDialog**
custom modal ภาษาไทย (Sprint 13d-P1) — มี title "⚠️ ยืนยันการทำรายการ"
+ ปุ่ม "ยกเลิก" (เทา) + "ยืนยัน" (สีแดง). ไม่ใช่ native browser confirm.

หลัง disable → row ยังโชว์ในตารางแต่ status เป็น "—" + action เปลี่ยน
เป็น "↺ เปิดใช้งานใหม่" (Sprint 13d-P4 + Sprint 13f-P2 สำหรับ WHT).

ถ้า user ไม่มีสิทธิ์ดู (เช่น accountant กับ /settings/api-keys) → ขึ้น
"ต้องมีสิทธิ์ผู้ดูแลระบบ" + shield icon (Sprint 13d-P2 NoAccessState).
ถ้าไม่มีสิทธิ์เขียน → ปุ่ม create/edit/disable ถูกซ่อน (Sprint 13d-P3
PermissionGate).

> (กดทีเดียวปิดเลย). เป็น UX nit ที่ flag ไว้สำหรับ housekeeping sprint
